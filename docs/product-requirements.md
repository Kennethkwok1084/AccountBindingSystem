# 需求基线与业务规则

本文档从 [完整深度研究报告](./deep-research-report.md) 中提炼第一版必须遵守的业务边界、流程和验收口径，供开发、测试和实施统一对齐。

## 1. 项目目标

系统目标不是单纯导入导出 Excel，而是稳定管理“移动 PPPoE 账号池”的完整生命周期，覆盖以下四条主线：

| 主线 | 目标 |
|---|---|
| 账号池管理 | 管理移动账号导入、分批次、可用性、禁用与恢复 |
| 收费清单处理 | 对收费记录做增量识别、预览、执行和导出 |
| 完整名单校正 | 用完整学生名单校正有效期并释放已到期绑定 |
| 台账与治理 | 处理批次预警、批量换绑、手动换绑和全链路审计 |

## 2. 范围边界

| 类别 | 包含项 | 不包含项 |
|---|---|---|
| 账号管理 | 仅管理移动账号 | 不管理联通、电信真实账号 |
| 文件处理 | `.xlsx` 导入导出 | 不做 OCR、PDF、图片识别 |
| 调度 | 预警、到期释放、待办生成 | 不做企业微信或短信推送 |
| 权限 | 单管理员 | 不做多角色 RBAC、SSO |
| 对接方式 | 导出 Excel 给现有后台导入 | 不直接调用现网后台接口 |

## 3. 固定业务规则

以下规则视为第一版验收基线，不应在实现时自行放宽：

| 规则项 | 固定要求 |
|---|---|
| 管理范围 | 仅管理移动账号 |
| 移动密码 | 固定为 `123123` |
| 批次有效期 | 普通批次默认一年；`0元账号` 长期有效 |
| 分配优先级 | 默认优先新批次，且普通批次优先于 `0元账号` |
| 收费处理顺序 | 按 `charge_time ASC`，只处理最早 20 人，其余报错 |
| 套餐有效期 | 包月 `31` 天；包年 `365` 天 |
| 校正规则 | 收费记录先推算，完整学生名单为最终校正依据 |
| 自动任务 | 每天凌晨执行 |
| 批次预警 | 提前 1 天预警次日到期批次 |
| 导出模板 | 列顺序固定，联通/电信列必须存在且为空 |
| 导出命名 | `移动YYMMDDHHmm.xlsx` |
| 部署环境 | Linux 内网 |
| 使用角色 | 单管理员 |

## 4. 默认配置与待确认项

以下项在原始需求中未完全指定，当前采用可落地默认值：

| 项目 | 当前默认值 | 说明 |
|---|---|---|
| 系统时区 | `Asia/Shanghai` | 写入系统配置 |
| 完整名单 Sheet | `Sheet1` | 允许后续配置化 |
| 低库存阈值 | 上线前配置 | 需求未写死 |
| 水位规则 | `charge_time > last_processed_charge_time` | 后续可升级为 `(time,row_hash)` |
| 批次到期处理 | 自动预警 + 人工确认换绑 | 第一版不做全自动换绑 |

## 5. 核心业务流程

## 5.1 收费清单处理

1. 上传收费清单 Excel。
2. 校验模板和列名，识别增量数据。
3. 按收费时间升序排序后分类为 `allocate / renew / rebind / fail`。
4. 页面先展示预览结果，不写业务数据。
5. 管理员确认执行后，按明细逐行事务处理。
6. 执行结果写入当前绑定、历史台账、操作批次与导出任务。

关键约束：

- 只处理水位之后的新收费记录。
- 只执行最早的 20 人。
- 无可用账号时必须明确失败原因。
- 执行接口必须幂等，避免重复提交。

## 5.2 完整名单导入

1. 上传完整学生名单。
2. 校验模板并计算“可校正记录、到期释放数、冲突数”。
3. 执行时仅校正有效期、释放已到期绑定。
4. 若名单与系统当前绑定不一致，只生成冲突预警，不自动覆盖当前绑定。

## 5.3 批次到期治理

1. 定时任务在 D-1 生成批次预警。
2. 管理员进入待办查看受影响学生。
3. 预览批量换绑结果。
4. 人工确认执行换绑。
5. 生成导出文件并沉淀审计记录。

## 5.4 手动换绑

1. 搜索学号并拉取当前绑定。
2. 选择新账号。
3. 指定旧号处理方式：`disable` 或 `release`。
4. 写入绑定变更与台账。
5. 立即生成单行导出文件。

## 6. 页面与模块清单

| 模块 | 目标 | 核心能力 |
|---|---|---|
| 登录与设置 | 管理员进入系统 | 登录、改密、配置维护 |
| 账号池管理 | 管理库存 | 导入、筛选、禁用、恢复、查看历史 |
| 批次管理 | 管理批次优先级 | 查看批次状态、到期日、库存分布 |
| 完整名单导入 | 校正有效期 | 上传、预览、执行、冲突提示 |
| 收费清单处理 | 增量处理收费数据 | 上传、预览、执行、导出 |
| 批次换绑 | 处理到期批次 | 预览影响学生、执行换绑 |
| 手动换绑 | 处理个别异常 | 选新号、旧号处理、单行导出 |
| 台账与审计 | 历史追溯 | 当前绑定、历史绑定、导入导出记录 |
| 预警中心 | 待办闭环 | 低库存、批次失效、失败任务处理 |

## 7. 验收标准

| 验收项 | 标准 |
|---|---|
| 账号唯一性 | 同一时刻同一移动账号不能绑定给两个学号 |
| 历史可追溯 | 能查任一学生的历史账号与任一账号的历史使用人 |
| 收费预览正确 | 增量筛选与分类结果可与人工核对一致 |
| 执行规则正确 | 严格遵守“最早 20 人优先处理” |
| 有效期正确 | 包月 31 天、包年 365 天，完整名单可校正 |
| 自动任务正确 | 每天凌晨执行，生成预警并释放到期绑定 |
| 导出模板正确 | 列顺序与列名完全匹配，空列必须存在 |
| 命名规则正确 | 导出文件名符合 `移动YYMMDDHHmm.xlsx` |
| 手动换绑可用 | 换绑成功且能生成单行导出 |
| 审计可查询 | 导入、导出、执行、调度、失败均可追溯 |

## 8. 开发注意事项

- 第一版优先保证规则正确，不追求自动化过度扩展。
- “完整名单冲突不自动覆盖”是重要保护规则，避免误改生产数据。
- `0元账号` 应通过批次优先级显式控制，避免长期号提前耗尽。
- 未来若取消“最早 20 人”限制，应改配置或规则层，不应直接改动底层数据模型。
