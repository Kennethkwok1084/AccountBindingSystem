# 系统架构设计

本文档提炼项目当前实现版的一线架构决策，便于前后端拆分开发和联调。研究论证、备选方案和历史 DDL 草案见 [完整深度研究报告](./deep-research-report.md)。

## 1. 技术选型

| 层级 | 技术 | 作用 |
|---|---|---|
| 前端 | Vue 3 + Vue Router + 自定义 CSS 中后台 | 页面、表单、表格、文件上传、预览执行、台账查询 |
| 后端 | Flask + SQLAlchemy + Alembic | 业务规则、导入导出、审计、配置管理、迁移 |
| 调度 | APScheduler 3.x 独立进程 | 预警、到期释放、临时文件清理 |
| 数据库 | PostgreSQL 16 | 事务、唯一约束、行锁、历史台账 |
| 部署 | Docker Compose + Nginx | 统一部署、反向代理、静态资源承载 |

## 2. 总体拓扑

推荐固定四服务部署：

| 服务 | 说明 |
|---|---|
| `postgres` | 持久化业务数据 |
| `backend` | Flask API 服务 |
| `scheduler` | 独立调度进程，和 backend 共用代码镜像 |
| `nginx` | 同域提供前端静态资源并代理 `/api` |

关键原则：

- `backend` 不启动定时器。
- `scheduler` 独立运行，避免重复执行。
- `nginx` 同域部署，优先使用 `HttpOnly` Cookie 维持会话。
- 生产环境使用 `.env` 提供数据库密码、管理员密码和会话密钥。

## 3. 后端模块边界

| 模块 | 职责 |
|---|---|
| `auth` | 登录、退出、改密、当前用户 |
| `dashboard` | 首页统计、最近任务、待办摘要 |
| `accounts` | 账号池、批次、禁用与恢复 |
| `students` | 学生资料、当前绑定、历史查询 |
| `operations` | 收费执行、完整名单执行、批次换绑、手动换绑、导入任务详情 |
| `exports` | 导出任务、下载 |
| `alerts` | 预警查询、处理闭环 |
| `scheduler` | 任务清单、运行日志、手工触发 |
| `audit` | 审计日志查询 |

配套建议：

- 规则计算放在 service 层，不直接散落在视图函数。
- 导入预览与执行共享模板校验器。
- 正式环境统一走 Alembic migration，不再使用 `db.create_all()` 建表。

- 执行类请求统一带 `X-Idempotency-Key`。

## 5. 一致性与并发原则

### 5.1 当前态与历史态分离

- `current_binding` 只保留当前有效绑定。
- `binding_history` 记录所有分配、续费、释放、换绑、禁用、人工修正。
- 任何改变账号归属的操作都必须同时更新当前态和历史态。

### 5.2 候选账号锁定

- 账号分配和换绑在事务内完成。
- 候选账号查询使用 `SELECT ... FOR UPDATE`。
- 预览阶段只给出建议账号，不冻结库存。

### 5.3 批处理模式

- 预览阶段无副作用。
- 执行阶段才写业务数据。
- 执行接口必须要求幂等键。
- 批处理明细通过 savepoint 隔离单行失败，避免整批脏事务蔓延。

## 6. 调度设计原则

| 任务名 | 作用 | 频率 |
|---|---|---|
| `batch_expire_warning` | 生成批次到期预警 | 每天凌晨 |
| `binding_expire_release` | 释放已到期绑定并补写历史台账 | 每天凌晨 |
| `inventory_alert_scan` | 生成低库存预警 | 每天凌晨 |
| `cleanup_temp_files` | 清理临时文件 | 每天凌晨 |

统一配置建议：

- `max_instances=1`
- `coalesce=True`
- 所有任务运行结果写入 `scheduler_run_log`

## 7. 关键架构决策

| 主题 | 决策 |
|---|---|
| 会话方式 | 同域部署 + `HttpOnly` Cookie |
| 配置存储 | 业务配置统一落到 `system_config` |
| 批量执行 | “先预览、后执行”，执行接口幂等 |
| 导出处理 | 所有导出都通过导出任务表追踪 |
| 审计策略 | 导入、导出、执行、下载、调度均记日志 |
| 文件存储 | 上传与导出文件落本地挂载卷 |

## 8. 研发协作建议

- 先冻结业务规则，再开始前后端并行开发。
- 后端优先完成模板校验、执行链事务、迁移体系。
- 前端优先完成登录、收费清单页、批次管理页、预警中心。
- 联调时优先覆盖“收费清单主链路”和“批次换绑闭环”。
