# 工程澄清与风险清单

本文档记录从 [完整深度研究报告](./deep-research-report.md) 进一步抽取出的高风险设计点和推荐落地口径，目标是减少开发返工。它不替代正式需求文档，而是作为研发协作时的补充决策单。

## 1. 有效期真值来源

当前设计同时出现了三个时间字段：

| 字段 | 含义 |
|---|---|
| `student.expected_expire_at` | 依据收费记录推算出的到期日 |
| `student.source_expire_at` | 依据完整名单校正后的到期日 |
| `current_binding.expire_at` | 当前生效绑定的到期日 |

推荐口径：

- `current_binding.expire_at` 作为“当前执行真值”。
- `student.expected_expire_at` 保留为收费推算结果，用于解释和对账。
- `student.source_expire_at` 保留为外部名单校正值，用于覆盖推算误差。
- 完整名单导入执行后，若确认学生仍处于有效绑定，应同步更新 `current_binding.expire_at`。
- 定时释放任务只依据 `current_binding.expire_at` 判断是否释放，避免多个口径并存。

## 2. 收费水位边界

当前默认规则是：

- 仅处理 `charge_time > last_processed_charge_time` 的记录。

风险：

- 若两批数据存在相同时间戳，可能漏单或重单。

推荐口径：

- 第一版按既定规则实现，但在文档和代码中明确标记这是已知风险。
- 为后续升级预留 `(charge_time, row_hash)` 或 `(charge_time, row_no)` 复合水位方案。
- 预览结果中回显“处理前水位”和“处理后水位”，便于运营核对。

## 3. 预览与执行的一致性

风险：

- 预览看到的候选账号，执行时可能已被其他操作或调度器占用。

推荐口径：

- 预览阶段只产生动作计划，不冻结账号。
- 执行阶段重新获取候选账号并加锁。
- 若执行结果与预览不一致，必须返回明确冲突原因，如“账号已被占用，请重新预览”。
- 前端执行成功后刷新批次详情，不直接复用旧预览当作最终结果。

## 4. 批次导入与批次元数据

风险：

- 账号池导入时，若批次不存在，系统如何获得 `batch_type`、`priority`、`expire_at`、`warn_days` 并不明确。

推荐口径：

- 第一版允许“导入时按文件携带的批次字段创建批次”。
- 若导入模板缺少关键批次元数据，则导入失败，不做隐式创建。
- 后台保留批次管理页面，允许后续单独维护优先级和到期日。

## 5. CSRF 与会话联调

风险：

- 总报告要求所有写接口带 `X-CSRF-Token`，但前端如何获取和刷新该值没有单独写清。

推荐口径：

- 登录成功时返回 `csrf_token`。
- 前端在内存态保存该值，并对所有写请求自动附带 `X-CSRF-Token`。
- 若收到 `401`，前端跳转登录页。
- 若收到 CSRF 失效错误，前端先调用 `/auth/me` 或重新登录获取新 token，再提示用户重试。

## 6. 初始化脚本与迁移脚本分离

风险：

- 总报告中的 DDL 草案包含 `DROP TABLE IF EXISTS`，适合初始化，不适合多人协作的演进式升级。

推荐口径：

- `schema-init.sql` 仅用于本地全新初始化。
- 正式开发引入迁移工具，增量维护 schema。
- 文档中明确“初始化脚本”和“生产迁移脚本”不可混用。

## 7. 并发必测场景

建议把以下场景升为集成测试必测项：

| 场景 | 目标 |
|---|---|
| 同一账号并发分配 | 验证唯一约束和行锁是否生效 |
| 同一收费批次重复执行 | 验证幂等键和状态冲突处理 |
| 调度释放与人工换绑同时发生 | 验证事务边界和冲突返回 |
| 预览后库存变化再执行 | 验证重新加锁与冲突提示 |
| 重复导入相同账号池文件 | 验证去重和错误明细 |
